# VihuPlanet Asset Pipeline

## Artist workflow

```
Artist
  ↓
Generate artwork
  ↓
Resize using GIMP
  ↓
Commit into raw/
  ↓
Push to main
  ↓
GitHub Action
  ↓
Normalize into production/
  ↓
Generate manifest.json for every collection
  ↓
Synchronize production/ (assets + manifests) to
dexprab/vihustudio/vihuplanet/world-library/
  ↓
Hero automatically renders new artwork
```

No manual registration. No code changes.

## Manifests

Every collection folder under `production/` (`skies/`, `story-homes/`,
`dreaming-home/`, `nature/trees/`, `nature/flowers/`, `nature/clouds/`,
`nature/rocks/`, `nature/shrubs/`, `nature/waterfalls/`, `decorations/`,
`companions/`) gets a `manifest.json` — a flat JSON array of PNG
filenames, nothing else:

```json
[
  "twilight-sky.png",
  "moonlit-sky.png",
  "storybook-night.png"
]
```

`manifest.json` is generated by `tools/generate_manifests.py` on every
run of the Asset Normalizer workflow. **Artists must never edit
manifests manually** — they're regenerated from whatever PNGs exist in
each collection every time production assets are synchronized, so a
hand edit is silently overwritten on the next push.

## Why manifests, not directory listings

VihuStudio's World Library provider used to discover assets by
requesting a folder URL (`world-library/skies/`) and parsing the
directory listing a dev server returns for it. GitHub Pages and most
static hosts disable directory browsing and return `404` for that
request, so production artwork was never discovered even though
synchronization was working correctly. Manifests fix this: the
provider fetches `world-library/skies/manifest.json` instead, which is
just another static file every host serves without issue.

## Running locally

```
pip install Pillow
python tools/normalize_assets.py
python tools/generate_manifests.py
```
